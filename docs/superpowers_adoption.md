# Superpowers Adoption Note

## 目的

この文書は、`obra/superpowers` の考え方を `youtube_patrol_v2` にどう取り入れるかをまとめたものです。

ここでの目的は、外部のワークフローをそのままコピーすることではありません。
このプロジェクトの既存ルールを壊さずに、次の改善だけを取り込みます。

- 先に調査する
- 小さく実装する
- 実装後に確認する
- 最後に判断をまとめる

## 取り入れるもの

### 1. 段階的に進める考え方

`superpowers` では、いきなりコードを書くのではなく、考えてから進める流れが重視されています。

このプロジェクトでは、次の 4 段階に整理して使います。

1. 調査
2. 実装
3. 検証
4. 統合

これは既存の運用文書とも相性がよく、変更の影響を追いやすくなります。

### 2. 役割を切り替えて考える方法

`superpowers` には、計画、実装、レビューのように視点を切り替える考え方があります。

このプロジェクトでは、1 人のエージェントが次の役割を順番に担当します。

- 調査担当
- 実装担当
- 検証担当
- 統合担当

これにより、作業中の見落としを減らしやすくなります。

### 3. レビュー視点を必ず残すこと

変更が小さくても、次の視点は最後に確認します。

- 既存仕様を壊していないか
- YouTube を直接売買シグナル扱いしていないか
- 日本語通知の読みやすさを落としていないか
- docker 前提の運用とずれていないか

## そのまま取り入れないもの

### 1. 重い手順の固定化

`superpowers` には便利な skill 群がありますが、このプロジェクトでは毎回すべての手順を重く回す必要はありません。

理由:

- 小さな修正が多い
- 既存の運用文書がすでにある
- 調査対象が比較的はっきりしている

そのため、軽い導入に留めます。

### 2. プロジェクト方針とぶつかる一般ルール

このプロジェクトには、すでに明確な判断基準があります。

例:

- YouTube は入口であり、最終判断ではない
- `BUY` は価格確認などの裏取りが必要
- AI は提案役に留める

外部ワークフローより、これらの既存決定事項を優先します。

## 優先する文書

仕様や運用判断で迷ったときは、次の順で確認します。

1. `DECISIONS.md`
2. `PROJECT_STATUS.md`
3. `NEXT_STEPS.md`
4. `docs/operation.md`
5. `docs/architecture.md`
6. `README.md`

`AGENTS.md` は作業の進め方を示す文書であり、売買判断や運用方針そのものを上書きするものではありません。

## 導入後の期待効果

- 変更前の確認漏れが減る
- 影響範囲を説明しやすくなる
- 実施した確認と未実施の確認を分けやすくなる
- 作業報告の形が安定する

## 現時点での制約

事実:

- `superpowers` の考え方は参考にできる
- ライセンス上、取り込みは可能

未確認:

- 外部 skill をそのままこの環境へ導入すること
- docker 権限付きでの常時運用確認

そのため、現時点ではまず運用ルールの導入に留めます。
